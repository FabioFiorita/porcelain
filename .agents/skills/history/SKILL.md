# Porcelain — decision history

The append-only audit trail. Each entry records **why** a decision was made; the
**what** (current behavior) lives in its home skill (`architecture`, `product`,
`audit`, `releasing`). The skill is the source of truth for how things work today;
this log is the chronology and the rationale.

## Maintenance rule

- **Dateless bullets, append-only.** Position is the chronology — git holds the
  real timestamps. Append at the end; match the terse style (bold lede for a major
  decision, GOTCHA/file pointers inline).
- Every entry also updates its **home skill** in the same commit (hard rule 4):
  the log records *why*, the skill states *what*.
- Never edit old entries to reflect new reality — write a new entry. The skills
  carry current truth; the log is immutable history.

- **Untracked-folder diff crash fixed** (`EISDIR: illegal operation on a directory, read` when clicking a newly-added folder in the Changes list). Root cause in `src/main/git.ts`: `gitStatus` ran `git status --porcelain=v1` at the default `-unormal`, which **collapses an untracked directory into a single `dir/` row**. That folder row entered the flow/changes list; clicking it sent the directory path to `gitDiffFile`, whose untracked branch calls `readFile(dir)` → EISDIR (blank tab + error). Fix: add `-uall` (`--untracked-files=all`) to both the `gitStatus` invocation and `gitDiffFile`'s status probe, so git lists each new file individually and every Changes row is a real, diffable file. Considered statting the path in `gitDiffFile` and expanding dirs on read, but that leaves a non-diffable folder row in a per-file review tool — wrong altitude; `-uall` fixes it at the source. `parseStatus` already handles `?? file` so no parser change. The flowCache key includes status output, so the extra rows bust the cache correctly. Perf note: `-uall` only changes *reporting* granularity (the default already recurses untracked dirs to classify them) — it's what most git GUIs do. Verified lint/typecheck/test(99)/build green.

- **Split-diff column-overlap bug fixed** (user report — split view rendered with long lines overrunning the opposite column, text colliding). Root cause in the shared virtualizer: `VirtualRows` wrapped every row in `w-max` (grow-to-content, correct for the single-column unified diff + source view where it drives one horizontal scroll). A split row is a `flex` of two `flex-1` `SplitCell`s, so under `w-max` the row sized to the **sum** of both columns' unwrapped (`whitespace-pre`) widths, the two halves were then forced equal, and the longer line overflowed its half with no clipping — bleeding over the neighbor. Fix: added a `fitWidth` prop to `VirtualRows` (true → `w-full` viewport width instead of `w-max min-w-full`, on both the sizer and each row); `HunksView` passes `fitWidth={diffMode === 'split'}` and each `SplitCell` gained `overflow-hidden`.
