# Porcelain architecture

## Stack

| Area | Decision |
|---|---|
| Shell | Electron via **electron-vite**, React 19, TypeScript (strict) |
| UI | **shadcn/ui on Base UI** (`@base-ui/react`, not Radix) + Tailwind CSS v4, `base-nova` preset, Geist font, dark mode default |
| Client architecture | **Porcelain's own conventions** (see "Client architecture" below): tab-store routing, domain data hooks, one public component per file |
| Client state | **zustand** — small stores per concern; no other state libraries |
| Git backend | Shell out to `git` CLI from the main process; parse porcelain-format output; no git libraries |
| Per-repo config | App-side JSON store under `~/Library/Application Support/porcelain`, keyed by repo path; never write into work repos |
| Package manager | **pnpm** |
| Lint/format | **Biome** (no ESLint/Prettier) |
| Tests | **Vitest** (unit/component, `src/**/*.test.{ts,tsx}`) + **Playwright** (Electron e2e in `e2e/`, `*.spec.ts`) |

## App shell

> Canonical names for these regions (Viewer, Quick Access, Sidebar, Top bar, sidebar tabs, tab kinds, overlays) live in the `## Nomenclature` glossary in `CLAUDE.md` — use those words.

- No repo open → full-screen `welcome` component (repo picker); shell renders only once a repo is selected.
- Layout: shadcn `sidebar` left + `SidebarInset` holding the tab bar and viewer + the right Quick Access sidebar. NO embedded terminal — Porcelain is a *companion to* the user's terminal (Ghostty/Warp), never a terminal itself; don't reintroduce one.
- **One repo per window** — window state is scoped to a single repo/worktree.
- Shell components live in `src/renderer/src/components/shell/` (`app-shell`, `app-sidebar`, `right-sidebar` + its `pinned-group`/`quick-commands-group`/`commit-group` sections, `file-tree` + `tree-node`, `tab-bar`, `viewer`); the viewer's pieces (`file-content`, `text-file-view`, `editor-source`, `source-view`, `source-context-menu`, `find-bar`, `use-path-actions`) live in `components/viewer/`; stores: `stores/tabs.ts`, `stores/repo.ts`.
- File tree: lazy per-directory reads (`readDir` on expand), nothing indexed up front; built from shadcn `SidebarMenu` + `Collapsible` + `ContextMenu`.
- Glass look: `vibrancy: 'under-window'` + `titleBarStyle: 'hiddenInset'` + transparent backgroundColor in main; dark theme `--background`/`--sidebar` have alpha; `.app-drag`/`.app-no-drag` classes mark window-drag chrome. The left sidebar's **title bar is continuous** (no vertical divider cuts through it — see the rail bullet): the macOS traffic lights own its left strip (they have priority), and the `ProjectSwitcher` is **start-aligned and fills the width between the lights and the show-hidden eye** — a `flex-1 justify-start` wrapper with the header `gap-0` so it spans all the way to the eye, and the trigger button itself is `w-full` so the name+chevron sit flush at the start (chevron hugging the name via `gap-1`, not pushed to the far edge). `pl-7` holds the start clear of the lights and the name `truncate`s as the rail narrows, so it yields instead of overlapping (this supersedes the old centered layout; the `pl-7` + `truncate` combo is what keeps the earlier left-aligned-collision problem from recurring). The chrome headers (rail top, panel header, main `TopBar`, right sidebar) are all `h-12` + `items-center` so their content shares one baseline; `trafficLightPosition: { x: 19, y: 24 }` in main positions the macOS buttons in that 48px bar (don't change one header height without the others, or the lights drift).
- Sidebar is drag-resizable: `sidebarWidth` in `stores/preferences.ts` (clamped 180–520) feeds `--sidebar-width` on `SidebarProvider`; handle in `sidebar-resize-handle.tsx`. During the drag the handle writes the CSS variable straight onto the `[data-slot="sidebar-wrapper"]` element and commits to the store only on mouseup — store writes per mousemove re-render the whole app.
- Right "Quick Access" sidebar (`right-sidebar.tsx`): a SECOND `SidebarProvider` nested inside `SidebarInset` (controlled via `rightSidebarOpen` preference; `shortcut="."` — the provider gained a `shortcut?: string | null` prop so the two providers don't both grab Cmd+B). `TopBar` lives inside the inner provider, so the left sidebar's toggle/state are captured by `RepoShell` (rendered between the providers, where `useSidebar()` is the outer one) and passed down as props. The two `TopBar` toggles use DELIBERATELY DIFFERENT icons — left = `PanelLeft` (file panel), right = `Zap` (quick actions) — never mirror-image panel icons. Sections follow the left sidebar's active tab (`sidebarTab` in `stores/preferences.ts`, driven by the left icon rail), each in its own file (`pinned-group.tsx`, `quick-commands-group.tsx`, `commit-group.tsx`): Files → `files-quick-access.tsx` (a vertical split — `pinned-group.tsx` rendering `TreeNode`s from `pinnedEntries` on top, a drag-resizable divider, then the `notes-card.tsx` quick-notes editor below; see "Notes card"); Changes + History + Feature → Quick commands; Changes + Feature → Commit composer (the Feature tab mirrors Changes — you review a feature to commit it) (`gitCommitConventions` = `parseConventions` in `src/main/conventions.ts` over the last 200 log subjects → type/scope chips + message; chips are STRICTLY history-derived — top 8 types/scopes by frequency, `DEFAULT_COMMIT_TYPES` only when the history has no conventional commits). "Commit" runs `gitCommit` (in git.ts: `git commit -m` on the **staged** changes only — no `git add`; git stderr rethrown so the mutation error renders inline); a separate "Stage all" button runs `gitStageAll` (`git add -A`) via `useStageAll` (inline confirmation/error in the composer); per-file staging lives in the **left** Changes list — `parseStatus` derives `staged`/`unstaged` booleans from the porcelain XY columns onto `ChangedFile` (flow through `FlowFile`), `changes-list.tsx` shows a dot per row (emerald = staged, amber = partially staged `MM`) and a right-click menu with **Open file** (opens the full working-tree file as an absolute-path `file` tab in the viewer — like the feature list — and flips the sidebar to Files via `setSidebarTab('files')`; the row's own click still opens the diff; hidden for deleted files) / Stage (`gitStageFile` = `git add -- path`) / Unstage (`gitUnstageFile` = `git restore --staged -- path`) via `useFileStaging`; all three staging hooks `utils.gitFlow.invalidate()` so the dots update. GOTCHA: the row's `ContextMenuTrigger` renders the `SidebarMenuButton` via the `render` prop (not as a wrapping child) — `SidebarMenuBadge`'s vertical position depends on the `peer/menu-button` sibling relationship, so wrapping the button in an extra trigger element drops the M/U badge to the bottom of the row. `useCommit` in `hooks/use-commit.ts` owns the gitFlow/gitLog/conventions invalidation. Quick commands run a WHITELIST (`QUICK_COMMANDS` in git.ts) via `useQuickCommand` — stdout+stderr combined (push logs to stderr), shown inline in the section, errors in red; the hook does a blanket `utils.invalidate()` after every run since pull/stash change everything (the one sanctioned blanket invalidation). Args are resolved server-side by the pure `quickCommandArgs(id, pullMode)` (git.ts, tested) — every entry is static except `pull`, which appends `--no-rebase`/`--rebase` per the `pullMode` preference (Settings → General "Pull strategy", default merge; read at call-time via `usePreferencesStore.getState()` and passed through the `gitQuickCommand` mutation) so the user's choice wins over their `pull.rebase` gitconfig. The whitelist stays authoritative — the only parameterized variation is that one flag.
- Pinning: per-repo `pinnedPaths` in the config schema, tree context menu Pin/Unpin (`pinPath`/`unpinPath`, `DirEntry.pinned` flag); invalidate `readDir` + `pinnedEntries` together.
- Notes card: per-repo `notes` string in the config schema (`notesFor`/`withRepoNotes` in `repo-config.ts` → `repoNotes` query + `setRepoNotes` mutation → `use-repo-notes.ts` hook). A TipTap WYSIWYG editor (`notes-card.tsx`) — markdown rendered live as you type, persisted as a markdown string via `tiptap-markdown`, with EditorSource-style debounced autosave (800ms + flush on unmount). SCOPED EXCEPTION to the "no CodeMirror/Monaco, file viewer stays plain-textarea" rule: TipTap is allowed for the notes card ONLY (it's a companion surface, not source editing). StarterKit v3 bundles Link (configure via StarterKit, don't add `extension-link` — double-registration throws); Placeholder is separate. Toolbar is shadcn `Toggle`/`Button`/`DropdownMenu`. `editor.storage.markdown` is typed via a `declare module '@tiptap/core'` augmentation (so `@tiptap/core` is an explicit devDependency) — no cast. The resizable divider is `NotesResizeHandle` (`sidebar-resize-handle.tsx`, vertical sibling of the width handles): drives `--notes-height` on `[data-slot="files-quick-access"]` live, persists `notesHeight` (preferences store, clamped 100–600). GOTCHA: ProseMirror calls `document.elementFromPoint` on mount — jsdom lacks it, so `test-setup.ts` stubs it (returns null) alongside `matchMedia`.
- Tree multi-select: cmd/ctrl-click toggles selection (`stores/selection.ts`); context menu shows "Hide N items" when a selection exists; batch runs hide mutations then one invalidate.
- Folder hiding: hidden paths are absolute, any depth (files or dirs), stored per repo in the app config (`src/main/repo-config.ts` pure logic + `config-store.ts` persistence at `userData/config.json` via the generic `json-store.ts` — atomic tmp+rename writes, corrupt files backed up to `.corrupt-*`, and `updateConfig(mutate)` serializes read-modify-write so concurrent mutations never drop; read-only callers still use `loadConfig`). `readDir` filters them in the MAIN process; `showHidden` mode returns them flagged (`DirEntry.hidden`) and the UI dims them. Hide/unhide via right-click context menu; eye toggle in sidebar header. Tree refresh = `treeVersion` bump in `stores/repo.ts`.
- Dev/prod isolation: in dev, main sets `userData` to `porcelain-dev` before any config read and `seedDevConfig` (`src/main/dev-config.ts`) seeds empty recents with `~/Code/porcelain-playground` — `pnpm dev` always opens the playground, never the user's repos.
- Welcome screen lists recent repos (max 10, `recentRepos` query; `openRepoPath` mutation). On startup the last repo auto-opens (`restoreLastRepo` in `stores/repo.ts`); welcome only shows if none/missing.
- Flow-ordered review: pure logic in `src/main/flow.ts` — `DEFAULT_LAYERS` ordered Pages→…→Data→Tests; `layerFor` uses deepest-matching-segment (not first-match; filename patterns like `\.spec\.`/`\.stories\.` naturally beat the containing directory — no special cases); `parseImports` + `resolveImport` (relative + alias-suffix heuristic) connect changed files; `buildFlow` groups in layer order. `gitFlow` procedure reads ≤200 changed files (≤1MB each) for edges. Per-repo `layers` override: edited in the settings dialog's Review flow section (`components/settings/settings-dialog.tsx` + `flow-layers-section.tsx`, gear pinned at the bottom of the icon rail; the dialog nests a `SidebarProvider` with `shortcut={null}` and a General section for diff/markdown/pull-strategy prefs) → `repoLayers`/`setRepoLayers` procedures (regex validated in zod) → `layersFor`/`withRepoLayers` in `repo-config.ts`. Changes list renders these groups with `connects` shown as "→ file" lines.
- Feature view: widens the change under review into the whole feature, rendered through the SAME flow grouping. Pure logic in `src/main/feature-view.ts` — `expandContext` walks one hop out from changed files along their RELATIVE imports (`resolveRelativeImport`, Set membership for speed) to pull unchanged files in as `context`; `buildFeatureView` unions changed + context + an optional agent review set into `FeatureFile`s tagged `changed`/`context`/`shipped` (git status always wins over a declared source), computes in-view `connects`, and groups via `layerFor`. The `featureView` procedure (`api.ts`, memoized like `flowCache` on status+numstat+layers+reviewSet) reads changed-file + context sources and overlays the review set. Shared types (`FileSource`, `ReviewSet`, zod schemas) live in `src/main/review-set.ts`. Two surfaces: (1) the **Feature sidebar tab** — the navigation list (`components/git/feature-list.tsx`, Cmd+4, `use-feature-view.ts`, peer of Files/Changes/History; owns the shared `SourceMarker`); `changed` rows open a diff tab, `context`/`shipped` rows open a file tab. (2) The viewer `feature` tab — the **inline reading surface** (`components/git/feature-view.tsx`, `use-feature-reading.ts`), **MCP-only**: it renders the whole feature as one flow-ordered document with just the relevant lines — diff hunks for `changed` files, heuristic **symbol slices** for `context`/`shipped` (only the definitions the in-view files import). Slicing is pure + tested in `src/main/feature-slice.ts` (`parseImportBindings` → `collectImportedSymbols` → `sliceSource`, with caps + a whole-file fallback); `buildFeatureReading` (in `feature-view.ts`) assembles it; the `featureReading` procedure (`api.ts`, memoized on the same key as `featureView`, sharing the `gatherFeature`/`buildFeatureFromGather` helpers) returns `null` when there's no review set so the baseline stays the list and the heuristic only runs on curated files. The surface flattens every file's rows into ONE `VirtualRows` (the `HunksView` pattern — `buildRows`, fixed 20px rows); it's opened by an "Open inline read" button shown in the sidebar list only when `fromAgent`. The sidebar list also has a two-step **"Clear agent set"** button (when `fromAgent`) → `clearFeatureReview` mutation → `clearReviewSet` (`review-store.ts`: the app's ONE write to the agent channel — atomic tmp+rename deleting the repo's entry, see the `audit` skill) → reverts to the baseline; `useClearFeatureReview` invalidates both surfaces. The label/tooltip deliberately scope it — it removes ONLY the agent overlay; your working-tree changes keep showing as the baseline (so on a dirty tree, clearing looks like "nothing happened" because the feature usually *is* your diff — see `history`). **Agent channel:** an agent feeds review sets via the standalone stdio MCP server in `src/mcp/` (`server.ts` + pure `protocol.ts`/`review-file.ts`), which writes `~/.porcelain/review-sets.json` (NOT a work repo, NOT userData — a plain `node` MCP process can't resolve Electron's userData; `PORCELAIN_REVIEW_SETS` overrides for tests/dev). Four tools: `set_feature_review` (replace), `add_review_files` (merge), `get_feature_review` (read back the stored set — name + per-source breakdown + files as JSON, so the agent verifies/round-trips an idempotent update/recovers after compaction; `describeReview` in `review-file.ts` is the pure formatter), `clear_feature_review`. `src/main/review-store.ts` reads that file (re-validated with zod); `review-watch.ts` watches its directory and pushes `feature-view` app-events so MCP writes live-refresh the open view. The MCP server is a second main build input (`electron.vite.config.ts`) emitting `out/main/mcp/server.js`; it imports only Node builtins so a non-Electron `node` can run it. **Distribution = a Claude Code plugin** (Settings → "Claude Code plugin"): `plugin-assets.ts` (pure, tested — manifest/marketplace/skill builders + install commands) + `plugin.ts` (writes a local marketplace to `~/.porcelain/plugin/`, COPIES the built `server.js` in so it's runnable even from inside app.asar, and best-effort runs `claude plugin marketplace add` + `claude plugin install` via a login shell with augmented PATH). The plugin bundles the MCP server (inline `mcpServers` → `${CLAUDE_PLUGIN_ROOT}/server.js`) and the `review-with-porcelain` skill. Procedures `pluginInfo` (dir + manual commands) / `installPlugin` (`PluginInstallResult`) → `use-plugin.ts` → `components/settings/plugin-section.tsx`.
- Explore (feature flow): read-only comprehension — seed a symbol/file, walk the import/reference graph, render it in the SAME reading surface. Pure walker in `src/main/feature-explore.ts` (+ tests): `parseImportLocals` (local→imported binding map), `relativeImportTargets` (file seed, one hop), `symbolReferenceTargets` (symbol seed — slice the symbol's body via `sliceSource`, match used identifiers against relative imports, resolve), and `walkExplore` (BFS, injected `readSource` so it's unit-testable, depth/file caps); `buildExploreReading` turns the nodes into a `FeatureReading` (all `context`, sliced to the reached symbols — empty ⇒ all exports). The `exploreFeature` procedure (`api.ts`) reads the seed + walked files on demand (outside the working tree, 1 MB/file cap, walk stays inside the repo via `resolveRelativeImport` against the git file list) and returns the reading. New `explore` tab kind (`stores/tabs.ts`, carries an optional `symbol`), rendered by `components/git/explore-view.tsx` via `use-explore.ts` (`staleTime` 60s — a snapshot, not live). The reading-surface rendering was extracted to `components/git/reading-surface.tsx` (`buildRows`/`ReadingRowView`/`ReadingSurfaceBody`), now shared by the MCP `feature` tab and `explore`. Two entry points, both through `usePathActions.exploreFlow` (the seed path is made repo-relative): the viewer selection menus (`source-context-menu.tsx` read-only + `editor-source.tsx` editable — BOTH, next to *Find references*: who calls it vs. *Explore flow* = what it uses) and the file-tree right-click (`tree-node.tsx`, file seed). Heuristic, no LSP: relative imports only (won't cross the alias/route-string seam), degrades to whole-file when a symbol isn't located.
- Worktrees/branch: `useBranch` (rev-parse, 5s poll) + `useWorktrees` (`worktree list --porcelain`, parser in diff.ts); `WorktreeSwitcher` in the content panel's footer (shadcn `dropdown-menu`) calls the repo store's `switchTo` — the ONE place repo switching lives (closes all tabs, then opens the path; `ProjectSwitcher` uses it too). GOTCHA: Base UI requires `DropdownMenuLabel` (GroupLabel) inside `DropdownMenuGroup` — outside one it throws MenuGroupContext missing.
- History: `gitLog`/`gitCommitFiles`/`gitCommitDiff` (parsers `parseLog`/`parseNameStatus`); sidebar History tab opens `commit` tabs rendered by `CommitView` (file list + per-file diff via shared `HunksView`/`DiffModeToggle`).
- Sidebar = **nested floating tile**: an **icon rail** + a **content panel** (`app-sidebar.tsx`). Two `collapsible="none"` `Sidebar`s sit inside one `Sidebar variant="floating" collapsible="icon"` whose inner is flipped to a row via `*:data-[sidebar=sidebar]:flex-row` (+ `overflow-hidden` to clip the panel when collapsed). The **rail** is `--sidebar-width-icon` wide — overridden to **3.5rem** on the left `SidebarProvider` in `app-shell.tsx` (shadcn default is 3rem; a compact icon column): an empty `h-12` drag strip, then the four **monochrome, icon-only** tabs (`SidebarMenuButton size-9 justify-center p-0`, active = `bg-sidebar-accent` pill via `isActive`, inactive = `text-muted-foreground` — NO per-tab `--ink-*` color anymore), then the settings gear in its `SidebarFooter`. The **vertical divider** (`border-r border-sidebar-border`) lives on the rail's **`SidebarContent`**, NOT the whole rail `Sidebar` — so it runs ONLY between the icons and the list (starts below the header rule, stops above the footer rule), and the title bar + footer read as one continuous strip across rail+panel. Moving it back onto the rail `Sidebar` would cut the divider through the chrome — don't. The **panel** (`flex-1`) holds the continuous-title-bar header (start-aligned, full-width `ProjectSwitcher` + eye — see "Glass look"), the active list (a plain `sidebarTab` switch over Files/Changes/History/Feature — **no shadcn `Tabs` here anymore**, scroll on a `min-h-0 flex-1 overflow-auto` wrapper), and the `WorktreeSwitcher` **right-aligned/trailing** in its footer (`SidebarMenuButton w-auto` + footer `items-end`). Both footers are `h-12` so the rail-gear and panel-branch `border-t` lines align. Each rail icon has an explicit `Tooltip` (name + `Kbd` shortcut) because the rail is always icon-only — the `SidebarMenuButton tooltip` prop only shows when collapsed, which wouldn't cover the expanded state. **Cmd+B collapses to the rail** (`collapsible="icon"`, not offcanvas); clicking any rail icon `setOpen(true)` so switching while collapsed re-reveals the panel (`selectTab`). The resize handle renders only when expanded; `--sidebar-width` drives the *total* width (rail fixed, panel `flex-1`). Inner rail+panel are `bg-transparent` so only the outer floating tile carries the alpha glass — stacking two `--sidebar` alphas would double-darken.
- Shortcut hints use the shadcn `Kbd` component (`ui/kbd.tsx`): TopBar toggle tooltips (⌘B / ⌘.), sidebar tab tooltips (⌘1/⌘2/⌘3/⌘4), context menus (⌘X/⌘C/⌘V, ⌘W on tab Close, ⌘⇧S on Open to the Side), empty-viewer ⌘P hint, editor Unsaved chip (⌘S). PATCH to ui/kbd.tsx: inside a `tooltip-content` the kbd uses `bg-foreground/10 text-popover-foreground`, NOT shadcn's default `text-background` — our tooltip is `bg-popover` (not inverted), so `text-background` was dark-on-dark and the shortcut rendered invisible. Re-apply if a preset overwrites `ui/`.
- Shortcuts: Cmd+W intercepted in main (`before-input-event`, blocks default window-close) → the dedicated `app-event` IPC push channel (`pipeAppEvents` in `src/main/ipc.ts` → `window.porcelain.onAppEvent`, consumed in `hooks/use-app-events.ts`, mounted in `AppShell`) → renderer closes active tab or window; Ctrl+Tab/Ctrl+Shift+Tab cycle tabs and Cmd+Shift+S splits the active tab to the side (`use-app-shortcuts.ts`, keyboard-only). Shortcut placement policy is tiered and deliberate — see "Client architecture".
- Tab bar: horizontal shadcn `ScrollArea` (the ui component takes an `orientation` prop); titles truncate at `max-w-40`; middle-click closes; right-click context menu = Close / Close Others / Close to the Left / Close to the Right / Close All / Open to the Side (bulk actions in `stores/tabs.ts`, anchor tab activated when the active tab is closed). `TabBar` takes a `paneIndex` — see "Split view" under Routing for the two-pane model.
- Crash visibility: `ErrorBoundary` wraps the app in `App.tsx` (render crashes show the stack, not a blank window); in dev, main pipes renderer `console-message` and `render-process-gone` to stdout — Claude reads them from the `pnpm dev` log to self-verify.
- Vite: `optimizeDeps.entries` covers `src/**/*.{ts,tsx}` so every `@base-ui/react/*` entry is pre-bundled. A dep discovered lazily mid-session re-optimizes, loads a second React copy, and crashes with "Invalid hook call".
- `readFile` returns a `FileView` union (text | image dataUrl | binary size) — viewer renders each.
- Text file tabs render `TextFileView` (`components/viewer/text-file-view.tsx`, header = relative path + actions). Text files are ALWAYS editable (no edit button/state): `EditorSource` (`components/viewer/editor-source.tsx`) = plain textarea over a Shiki-highlighted backdrop with a line-number gutter (textarea text transparent + `caret-foreground`, aria-hidden mirror of memoized `CodeLine`s; keep textarea `pl-14` aligned with the gutter or the caret drifts). Both layers live in ONE `overflow-auto` scroll container — the textarea sizes to content via `field-sizing-content` so the container (not the textarea) scrolls, and the backdrop is `absolute inset-0` behind it. They CANNOT desync because there's a single scroll; the old design scrolled the textarea and synced the backdrop in `onScroll`, which lagged and left the native selection over the wrong lines after a scroll. Autosave: 800ms debounce after last keystroke + flush on unmount + Cmd+S; status chip bottom-right (Unsaved ⌘S / Saving… / error); saves via `useWriteTextFile` (`hooks/use-files.ts`, which owns the readFile + gitFlow + gitDiffFile invalidation); first keystroke pins a preview tab. Files over 5000 lines (`EDITABLE_MAX_LINES`) fall back to the read-only virtualized `SourceView`; markdown reader mode is read-only (source mode edits). The editor's own context menu covers Cut/Copy/Paste/Find references (textarea selection) + path actions; capture the selection in `onOpenChange` into state — nothing re-renders on text selection, so reading it at render time leaves items stale/disabled. Menu items show `Kbd` shortcuts inside `ContextMenuShortcut`. `FileContent` is keyed by path in `Viewer` so edit state can't leak across tabs. Deliberately NOT an editor: no CodeMirror/Monaco, no LSP.
- Viewer context menu (`components/viewer/source-context-menu.tsx`): Copy / Find references (selection) / Copy path / Copy relative path / Reveal in Finder (`useRevealInFinder` → shell.showItemInFolder). GOTCHA: the ui `ContextMenuTrigger` defaults to `select-none` — pass `select-text` or text selection silently breaks. Selection is captured in the menu's `onOpenChange`. The edit-mode textarea gets its own Cut/Copy/Paste menu (clipboard API + manual splice).
- Find references: `searchText` procedure = `gitGrep` (`git grep -n -I --untracked --fixed-strings`, ≤500 matches, exit 1 → []; parser `parseGrep` in diff.ts). Results render in a `search`-kind tab (`SearchView`); clicking a match opens the file tab with `Tab.line`, which `VirtualRows`' `scrollToLine` centers and `SourceView` highlights. `openTab` on an existing id updates `line` so repeated jumps work.
- Markdown files get a Reader/Source toggle (private `MarkdownModeToggle` in `components/viewer/text-file-view.tsx`; `markdownMode` preference, reader default). Reader = `MarkdownView` (react-markdown + remark-gfm, `prose prose-invert` via `@tailwindcss/typography`, loaded with `@plugin` in main.css); links get `target="_blank"` so main's `setWindowOpenHandler` opens them externally — gated by `isSafeExternalUrl` (`src/main/external-url.ts`, http/https/mailto allowlist; extend `ALLOWED_PROTOCOLS` deliberately, never drop the guard). Reader view is NOT virtualized — markdown only; never route code files through it.
- UI preferences (diff mode, markdown mode, pull strategy, sidebar tab, right/left sidebar open + widths, `pluginInstalled` — set true after the first successful Claude Code plugin install so the Settings CTA demotes to a quiet "Reinstall") persist via `zustand/middleware` `persist` to localStorage (`porcelain-preferences`). Both sidebars drag-resize (`SidebarResizeHandle`/`RightSidebarResizeHandle` write `--sidebar-width` directly during drag, commit on mouseup).
- Git: pure parsers in `src/main/diff.ts` (porcelain-z status, unified diff → `DiffHunk[]`, synthesized add-diff for untracked) + shell-out in `src/main/git.ts`. Sidebar has Files/Changes tabs (shadcn `Tabs`); changes list opens `diff` tabs; `DiffView` (`components/git/diff-view.tsx`) renders unified or split via `HunksView` (`hunks-view.tsx`, split rows paired del↔add in `toSplitRows`) and `DiffModeToggle` (`diff-mode-toggle.tsx`, preference in `stores/preferences.ts`) — both shared with `CommitView`.

## Packaging, signing, updates

- `electron-builder.yml`: appId `com.fabiofiorita.porcelain`, productName Porcelain, mac targets dmg + zip (arm64; the ZIP is what electron-updater downloads), hardened runtime, signs with the keychain "Developer ID Application" identity automatically. `notarize: false` until notarytool creds are configured (`APPLE_ID`, `APPLE_APP_SPECIFIC_PASSWORD`, `APPLE_TEAM_ID` env vars — then flip to true).
- Scripts: `pnpm dist` = typecheck + build + signed DMG/ZIP into `dist/` (no publish); `pnpm release` = same + publish to GitHub releases `fabiofiorita/porcelain` (requires `GH_TOKEN` env, bump `version` in package.json first).
- Auto-update: `electron-updater` wired in `src/main/updater.ts` — no-op unless `app.isPackaged`; checks on launch + every 4h, auto-downloads, installs on quit. ## Repo facts

- Aliases `@renderer/*` → `src/renderer/src/*` and `@main/*` → `src/main/*`, defined in FOUR places that must stay in sync: `electron.vite.config.ts`, `tsconfig.web.json`, root `tsconfig.json` (needed by the shadcn CLI), `vitest.config.ts`. `@main` imports in the renderer are **type-only** (`import type`) — never runtime-import main code; esbuild erases them so nothing leaks into the bundle.
- shadcn components: `src/renderer/src/components/ui/` (excluded from Biome); add via `pnpm dlx shadcn@latest add <name>`. Base UI uses `render` prop, not Radix's `asChild` — see `.agents/skills/shadcn/rules/base-vs-radix.md`.
- Tailwind/theme entry: `src/renderer/src/assets/main.css` (imports `shadcn/tailwind.css` and Geist).
- Main process = OS/git/fs access. Renderer = pure UI, no Node APIs.
- **Data fetching = TanStack Query via domain hooks** (`@trpc/react-query@11` + `@tanstack/react-query@5`, paired with tRPC v11). Components NEVER import `@renderer/lib/trpc` (Biome `noRestrictedImports` override enforces this in `components/**`); all server access goes through hooks in `src/renderer/src/hooks/use-<domain>.ts` that wrap `trpc.*.useQuery/useMutation` and OWN their post-mutation invalidation. Provider in `lib/query.tsx` (staleTime 30s, no focus refetch). The vanilla `trpcClient` is sanctioned only in zustand stores (`stores/repo.ts`) and the `use-app-events.ts` hook. React-query v5 idioms: mutations expose `isPending` (not `isLoading`); "keep last data while refetching" is `placeholderData: keepPreviousData` (import `keepPreviousData` from `@tanstack/react-query`), not `keepPreviousData: true`; query-level `onSuccess`/`onError` were removed (invalidation lives in mutation `onSuccess`). Never ad-hoc useEffect+useState fetching; never prefix invalidation/prefetch with `void` — use `async`/`await` or `await Promise.all([...])`.
- **Virtualized code rendering**: file viewer and diff view render through `VirtualRows` (`components/viewer/virtual-rows.tsx`, @tanstack/react-virtual, fixed 20px rows) — NEVER render all lines of a file; only mounted rows render to the DOM (Shiki tokenizes the whole file once, memoized — see "Syntax highlighting"). Diff hunks flatten to a `DiffRow` union (`header | unified | split`) before virtualization. Rows default to **grow-to-content** width (`w-max`, single horizontal scroll — unified diff, source view); the `fitWidth` prop switches to **viewport** width (`w-full`) for fixed multi-column layouts where each column owns a width share and clips its own overflow. Split diff passes `fitWidth` (each `SplitCell` is 50% + `overflow-hidden`) — without it, `w-max` sizes the row to the *sum* of both columns' unwrapped widths and the `flex-1` halves let the longer line overrun the other (overlap bug). Long lines clip in split; toggle to unified to scroll them.
- Query freshness: fs-backed queries keep the 30s default staleTime; git-backed queries are live (`gitFlow`: staleTime 0 + 3s poll while mounted; `gitDiffFile`: staleTime 0, `placeholderData: keepPreviousData`). The 3s `gitFlow` poll is cheap because main memoizes the flow result on a status+numstat+layers key (`flowCache` in `api.ts`) — changed-file contents are re-read only when the working tree actually changes.
- Hover prefetch: changes-list rows prefetch `gitDiffFile` (staleTime 2s) and file-tree rows prefetch `readFile` on mouseenter, so opening feels instant.
- **Syntax highlighting = Shiki**, theme `dark-plus`, singleton in `lib/highlight.ts` (`languageFor(path)` ext→lang map). Tokenization is **whole-file, not per-line**: `tokenizeLines(code)` runs `codeToTokensBase` over the entire content so grammar state carries across line breaks — per-line tokenization lost it and mis-colored multiline comments/template-literal continuation lines as code. `CodeLine` (`components/viewer/code-line.tsx`) is now a dumb renderer taking a precomputed `tokens: ThemedToken[] | null` for one line (plain-text fallback when null); its companion `useTokenizedLines(content, lang)` hook memoizes the whole-file token array (indexed by line) and feeds the file viewer + editor backdrop. The diff (`hunks-view.tsx`) can't see outside its hunks, so `tokenizeHunks` reconstructs each hunk's old (context+del) and new (context+add) image as contiguous text and tokenizes those — multiline state is correct *within* a hunk, cross-hunk context is inherently unavailable. Geist Mono's coding ligatures are disabled globally (`font-variant-ligatures: none` on `html` in main.css) so `===`/`!==`/`=>`/`??` stay legible instead of merging into a bar.
- **IPC = tRPC over a custom Electron link we own** (the only IPC pattern). tRPC **v11** paired with `@tanstack/react-query` **v5**. `electron-trpc` is gone (abandoned at 0.7.1, never supported v11) — we own the transport: the renderer client uses tRPC's official `httpBatchLink` with a custom `fetch` (`src/renderer/src/lib/trpc.ts`) that serializes each call to an HTTP request and ships the bytes over `window.porcelain.trpc` → `ipcRenderer.invoke('trpc')`; the preload (`src/preload/index.ts`) exposes that `porcelain` bridge via `contextBridge`; main (`src/main/ipc.ts`, `registerTrpcHandler`, registered once in `index.ts`) rebuilds the `Request` and replays it through tRPC's official `fetchRequestHandler` (endpoint `/trpc`, empty context) — ALL protocol logic (batching, input/error encoding) stays inside tRPC; we only shuttle bytes, so there are no internal-API assumptions to rot. Renderer clients: `createTRPCReact` (the `trpc` hooks proxy) + `createTRPCClient` (the vanilla `trpcClient` for stores), both over the same link factory — per-call `invoke` promises can't cross-talk, so two clients are safe. Router + procedures in `src/main/api.ts` (zod inputs, exported `AppRouter` + shared types like `RepoInfo`/`DirEntry`); `import type { AppRouter }` from main — type-only, no runtime coupling. Never use raw `ipcMain`/`ipcRenderer` for data (the ONE sanctioned raw channel is the `app-event` push), never cast (`as unknown as` is banned repo-wide). GOTCHA: the `httpBatchLink` URL (`http://localhost/trpc`) host is a throwaway — the request never leaves the process; and the bridge needs `Request`/`Response`/`fetch` globals, which Electron 42 (Chromium renderer + Node 24 main) provides on both sides.

- Tabs: single-click in the tree opens a *preview* tab (`Tab.preview`, italic title) that the next preview replaces; double-click (tree or tab) or entering edit mode pins via `pinTab`. Cmd+1/2/3/4 switch the sidebar tabs (`use-app-shortcuts.ts`).
- Cmd+F find-in-file: `FindBar` (`components/viewer/find-bar.tsx`, line-based case-insensitive matches, Enter/Shift+Enter cycle, Esc closes); current match feeds `SourceView`'s `highlightLine`. It is a FLOATING overlay (`absolute right-3 top-3 z-30` rounded card over the code), not a stacked bar — `TextFileView` wraps the content area in `relative` and renders it inside, so it never adds a third header row. Source rows never wrap (`whitespace-pre` in `CodeLine`, `w-max min-w-full` rows in `VirtualRows` → horizontal scroll); wrapping inside 20px virtual rows overlaps lines.
- `readFile` stats first and returns `{type:'too-large'}` above `MAX_READ_BYTES` (10MB, `src/main/read-limits.ts`) — never read the bytes of oversized files.
- Sidebar header = `ProjectSwitcher` dropdown (recent repos + open dialog; switching clears tabs). File rows render colored per-filetype icons from `components/viewer/file-icon.tsx` (`FileTypeIcon`/`FolderIcon`); `.DS_Store` is filtered in `readDir`.
- Git suggestions: pure `parseSuggestions` in `src/main/suggestions.ts` (status --porcelain=v2 --branch + stash list) → `gitSuggestions` query → sparkle rows above Quick commands in the right sidebar (5s poll).
- Theme: shadcn preset `b2D0xPJT8` (luma/Geist Mono/small radius), retinted to a **neutral graphite** accent — `--primary`/`--ring`/`--sidebar-primary`/`--chart-*` are monochrome (no colored brand accent), the single theme for v1. There is NO theme picker (a blue option + Theme settings tab were built and then cut — user wanted one dark theme to ship; see the `history` skill). ## Client architecture

The renderer has exactly one architecture; every feature follows it. Layering, top to bottom:

```
main process (src/main/api.ts procedures, pure logic in own modules)
  → lib/trpc.ts (ONE client; import restricted to hooks/ and stores/)
    → hooks/use-<domain>.ts (domain data hooks: queries, mutations, invalidation)
      → components/<area>/*.tsx (UI only; consume hooks + stores)
stores/ (zustand: client-only state — tabs, repo, preferences, selection)
```

### Routing — the tabs store IS the router

There is no URL routing and no router library; the active tab's `(kind, path, line)` in `stores/tabs.ts` is the entire navigation state. A screen = a `TabKind` (`file | diff | commit | search | feature | explore`). `Tab.path` is overloaded per kind (file path / file path / commit hash / query / — / seed file path — documented on the type; `explore` also carries an optional `symbol`). Tab ids are ALWAYS built with `tabId(kind, key)` from `stores/tabs.ts` — never hand-build `"diff:..."` strings. `Viewer` (`components/shell/viewer.tsx`) dispatches kind → view with an exhaustive `switch` (no default; the annotated return type turns a missing case into a compile error). Preview semantics: single-click opens a preview tab (italic) that the next preview replaces; double-click, editing, or a non-preview re-open pins (`pinTab`).

**Split view = panes, not extra tab state.** The store holds `panes: Pane[]` (a `Pane` is `{ tabs, activeTabId }`) plus `activePaneIndex`; the viewer is one pane unsplit, two side-by-side when split. The key invariant: **`openTab`/`pinTab`/`cycleTab`/`closeAllTabs` keep their signatures and always act on the active pane**, so every opener (file tree, finder, changes/history/search/feature) is pane-agnostic and needed zero changes. `openTabToSide(tab)` opens in the *other* pane (creating the second pane and focusing it). Pane-scoped ops — `closeTab`/`closeOtherTabs`/`closeTabsToLeft`/`closeTabsToRight`/`activateTab` — take `(paneIndex, id)`; `setActivePane(i)` focuses a pane (also done on tab click / pane mousedown). Closing a pane's last tab **collapses the split** (`normalize` drops empty panes, keeps ≥1; an empty single pane shows the welcome view). `pinTab(id)` pins the tab in whichever pane(s) hold it. Layout: **unsplit is unchanged** — the tab bar stays in the chrome `TopBar` (`<TabBar paneIndex={0}/>`); when split, the `TopBar` bar is replaced by a drag spacer and each pane renders its own `<TabBar paneIndex={i}/>` header inside the viewer. The divider is `SplitResizeHandle` (`sidebar-resize-handle.tsx`, same direct-CSS-var-during-drag trick) driving `--split-left` on `[data-slot="viewer-split"]`; ratio persists as `splitRatio` in preferences (clamped 0.2–0.8). Trigger = **"Open to the Side"** context-menu item on a file-tree row (`tree-node.tsx`) and on a tab (`tab-bar.tsx`), or **Cmd+Shift+S** on the active tab (`use-app-shortcuts.ts`, matched by `e.code === 'KeyS'` so it's layout-independent; splits the active pane's active tab to the side). Active pane shows a subtle accent on its tab-bar header border.

**Recipe — adding a new screen/tab kind** (e.g. `blame`), in order:
1. Pure logic in its own `src/main/<thing>.ts` module with a `<thing>.test.ts` next to it.
2. Procedure on the router in `src/main/api.ts` (zod input; `AppRouter` updates automatically).
3. Hook in `src/renderer/src/hooks/use-<domain>.ts` wrapping the procedure (new file only for a genuinely new domain).
4. `'blame'` added to `TabKind` in `stores/tabs.ts`.
5. View component `components/git/blame-view.tsx` — one public component, takes its key as a single prop (`hash`/`path`), reads data via the hook.
6. Opener calls `openTab({ id: tabId('blame', key), ... })`.
7. New `case` in `Viewer`'s switch (the compiler forces this).
8. Keyboard binding in `use-app-shortcuts.ts` if needed.

Repo switching is one store action: `useRepoStore.switchTo(path)` (closes all tabs, opens the repo) — used by `ProjectSwitcher` and `WorktreeSwitcher`; never clear tabs ad hoc. The right sidebar deliberately follows `preferences.sidebarTab` (the LEFT sidebar's tab), not the active main tab — two parallel nav axes by design.

### Data hooks (`src/renderer/src/hooks/`)

- One module per domain: `use-repo`, `use-files`, `use-git-flow`, `use-feature-view`, `use-diff`, `use-history`, `use-commit`, `use-worktrees`, `use-repo-layers`, `use-search`, `use-updates`, `use-app-events`. Thin declarative wrappers — no business logic.
- Query options (enabled guards `repo !== null`, `staleTime`, `refetchInterval`, `placeholderData: keepPreviousData`) live in the hook, not the component.
- **Hooks own invalidation**: each mutation hook lists its targeted invalidations in `onSuccess` (`Promise.all`). The ONLY blanket `utils.invalidate()` is `useQuickCommand` (pull/stash change everything — documented escape hatch). Hover prefetch is a hook too (`useDiffFilePrefetch`, `useReadFilePrefetch`).
- `use-app-events.ts` consumes the one main→renderer push channel (`window.porcelain.onAppEvent`), mounted once in `AppShell`. There are NO tRPC subscriptions — push rides the dedicated `app-event` IPC channel (the custom link is request/response only). Events: `close-tab`, `update-status`, `feature-view` (the last invalidates the `featureView` query when the agent channel file changes).
- Enforced: a Biome `overrides` block in `biome.json` makes importing `@renderer/lib/trpc` from `components/**` a lint error.

### State placement — one rule per kind of state

- Server/git/fs state → TanStack Query via domain hooks, nowhere else.
- Cross-component UI state → a zustand store in `stores/`, one file per concern; components subscribe with fine-grained selectors (`useXStore((s) => s.field)`) directly at the leaf — no prop-drilling stores (the one exception: `LeftSidebarHandle` drilled `RepoShell`→`TopBar`, forced by nested SidebarProviders).
- Prefs that survive reload → the single persisted `preferences` store (localStorage `porcelain-preferences`). Nothing else persists.
- Everything else → component-local `useState`. Never `useState` for state another component reads.
- Store actions may call other stores via `useXStore.getState()` (e.g. `switchTo` → `closeAllTabs`); components use hooks.

### Component authoring

- **One public component per file**; filename = kebab-case of the export. Private module-scoped subcomponents are allowed only when tightly bound to the export (a row, a toggle used once); an independent feature (own queries/state) gets its own file. Allowed co-location exceptions: inseparable variant pairs (`file-icon.tsx`, `sidebar-resize-handle.tsx`), a component plus its companion hook (`code-line.tsx` + `useHighlighter`), and mutually recursive components (`tree-node.tsx`: `TreeNode`/`DirNode`).
- Named exports only (`export function PascalCase()`); the sole default export is `App`. Class components only where React requires it (`ErrorBoundary`).
- Explicit return types: `React.JSX.Element` (`| null` when conditionally empty); handlers `void`/`Promise<void>`.
- Props typed INLINE in the destructuring parameter. A named `XProps` interface is allowed ONLY for generic components (`VirtualRowsProps<T>`). Domain/data types may be named interfaces.
- Handlers named by intent (`run`, `save`, `switchTo`, `select`) — never `handleX`; callback props use React-native `onX` names.
- Composition: plain prop-driven components + zustand; `children`-wrapper components for menu/boundary shells; render-prop only for generic virtualized lists (`VirtualRows`); Base UI's `render={<.../>}` slot to merge shadcn triggers. NO app-authored React context, no boolean-prop variant proliferation. (The former vercel-composition-patterns pointer is retired — these rules replace it.)
- Pure single-file helpers stay module-scoped at the top of the file; pure shared helpers go in `@renderer/lib/` with a test; reused stateful logic is a `useX` hook returning named callbacks. Derive, don't store, computed values.
- Always `cn()` for conditional/merged classNames — no template-string class concatenation.

### Keyboard shortcuts — tiered ownership (deliberate, don't "centralize")

1. Main-process `before-input-event` ONLY to override an Electron/OS default (Cmd+W).
2. App-global bindings that act on stores → `use-app-shortcuts.ts` (Ctrl+Tab, Cmd+1/2/3/4, Cmd+Shift+S).
3. A shortcut toggling one component's local state registers its own window listener in that component (Cmd+P in `file-finder`, Cmd+F in `text-file-view`).
4. Focused-element shortcuts as element `onKeyDown` (Cmd+S on the editor textarea).
5. Sidebar toggles via the `SidebarProvider` `shortcut` prop (Cmd+B / Cmd+.).

### Testing

- Pure logic (main-process parsers, lib helpers, zustand stores) → unit tests next to source (`foo.test.ts`), named after the unit. This is where most coverage lives — keep logic pure and main-side.
- Component tests (`foo.test.tsx`, next to source) mock the **domain hooks**, never the tRPC proxy: `vi.mock('@renderer/hooks/use-history', ...)` returning plain objects. Exemplars to copy: `components/git/history-list.test.tsx`, `components/git/changes-list.test.tsx`. Shape mock data with `@main` types so drift breaks the build.
- `lib/trpc.ts` is now import-safe under vitest (the custom link only touches `window.porcelain` when a request actually fires, not at module load), but component tests STILL mock the domain hooks, never the tRPC proxy — the hooks layer is the seam, and mocking hooks keeps tests free of the transport entirely.
- Setup: `src/test-setup.ts` (registered in `vitest.config.ts` `setupFiles`) wires jest-dom matchers, an explicit `afterEach(cleanup)` (globals are off — tests import from `'vitest'`), and a `window.matchMedia` stub (required by any test mounting `SidebarProvider`). jest-dom types load via the triple-slash reference in `env.d.ts`.
- Reset zustand between tests with `useXStore.setState(...)` in `beforeEach`. No snapshot tests.
- **Playwright Electron e2e** (in `e2e/`, run with `pnpm test:e2e` — builds first, then `playwright test`). This is the SECONDARY tier: most coverage stays in Vitest; e2e is a small set of high-value full-app flows + screenshot baselines, and is a **release gate** (the `releasing` skill), NOT part of the per-commit gate (hard rule 3 stays the four fast commands). Vitest only sweeps `src/**/*.test.*`, so the e2e `*.spec.ts` never collide. How it works: each test `_electron.launch`es the BUILT app (`out/main/index.js`) so the real preload is present and the tRPC/git data layer works — driving the dev URL in a bare Chromium does NOT (the renderer's data layer only exists via Electron's preload). The shared fixture (`e2e/helpers/electron.ts`) creates a deterministic git repo (`e2e/helpers/fixture-repo.ts` — a page/component/doc with a two-commit history + uncommitted work, fixed author/dates), seeds a temp `userData` (the launched build runs as `is.dev`, so main appends `-dev` to the `--user-data-dir` — write `config.json` THERE), points `PORCELAIN_REVIEW_SETS` at a temp file (so the Feature tab is deterministic + the user's real `~/.porcelain` is untouched), sets **`PORCELAIN_E2E=1`** (the runs are effectively **headless** — gated in `src/main/index.ts`: skip `mainWindow.show()`, `backgroundThrottling:false` so the never-shown window keeps painting for screenshots, `app.dock?.hide()`; Playwright drives the renderer over CDP so no OS window ever pops on screen — macOS Electron has no `headless:true` flag, this is the equivalent), and exposes `page` + nav helpers (`waitForShell`/`selectTab`/`openSettings`). Selectors are role/label based (rail tabs = `aria-label` Files/Changes/History/Feature; gear = `aria-label="Settings"`). Welcome-screen case: seed recents with a NON-EXISTENT path — a non-empty list stops the dev seed from auto-adding `~/Code/porcelain-playground`, then the `recentRepos` query prunes the dead path → genuinely empty → Welcome (deterministic on any host/CI). GOTCHAs: (1) **`PLAYWRIGHT_FORCE_ASYNC_LOADER=1` is REQUIRED** (baked into the `test:e2e` scripts) — on Node ≥22.15 Playwright 1.61 registers SYNC loader hooks (`module.registerHooks`) whose `resolve` context has a non-array `conditions`, so it throws `context.conditions?.includes is not a function` on the first relative TS import; the flag forces the old async `module.register` path. (2) Playwright is pinned to `e2e/tsconfig.json` (a self-contained, `paths`-free config) via the config's `tsconfig` field, kept separate from the four-config `@renderer`/`@main` alias set. (3) A worker fixture's first arg must be `({}, use)` (Playwright requires the destructuring pattern) — suppressed with a scoped `biome-ignore lint/correctness/noEmptyPattern`. (4) Screenshots are **DOM-only** (no native window chrome / vibrancy void — those still need computer-use), per-platform (`-darwin`); baselines live in `e2e/*.spec.ts-snapshots/` (committed); regenerate after intentional UI changes with `pnpm test:e2e:update`. `test-results/`, `playwright-report/`, `e2e/.artifacts/` are gitignored.

## Conventions

- **shadcn primitives only**: never hand-roll a primitive (sidebar, tabs, dialog, tree, …); search shadcn/registries first; building a new primitive requires user approval.
- Strict TS, no `any`, no `as unknown as`, no dead code, no commented-out code.
- Conventional Commits (`feat:`, `fix:`, `chore:`, `refactor:`, `test:`, `docs:`).
- Verification gate before any commit: `pnpm lint && pnpm typecheck && pnpm test && pnpm build` must all pass.
- **Decision history** lives in the `history` skill (moved out of CLAUDE.md — it's the *why* audit trail, loaded on demand, not always-on): dateless bullets, append-only, position = chronology (git holds timestamps). Terse style — bold lede for a major decision, GOTCHA/file pointers inline. Every entry also lands in its home skill (hard rule 4): the log records *why*, the skill states *what* — the skill is the source of truth for current behavior, the log is the audit trail.
- **Related skills**: `audit` = the security/correctness/perf/type invariants to preserve (read before main-process/IPC/config/git/packaging changes); `releasing` = the release runbook; `history` = the decision chronology; `product` = what/why for features. This skill is the *what* for code structure.
