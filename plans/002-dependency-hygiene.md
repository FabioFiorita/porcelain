## Why this matters

Two dependency-placement problems, one with a real latent visual bug:

1. **`next-themes` is dead wiring AND misplaced.** It sits in `dependencies` but
   its only importer is `src/renderer/src/components/ui/sonner.tsx`, a renderer
   component. No `ThemeProvider` is mounted anywhere in the app (the app is
   hardwired dark via `<html class="dark">` in `index.html`, with no theme
   picker). So `useTheme()` always returns the default `"system"`, which sonner
   passes to its `theme` prop. On an OS in **light mode**, sonner then renders
   **light toasts inside a dark-only app** — a visible inconsistency. The fix is
   to hardcode `theme="dark"` and remove the package entirely.

2. **Font packages are split inconsistently.** `@fontsource-variable/jetbrains-mono`
   is in `dependencies` while its two siblings `@fontsource-variable/geist` and
   `@fontsource-variable/geist-mono` are in `devDependencies` — yet all three are
   renderer-only CSS `@import`s. Per the packaging invariant (renderer-only libs
   belong in `devDependencies`; electron-builder copies `dependencies` whole into
   `app.asar`), jetbrains-mono is bloating the packaged bundle for no reason and
   contradicts its own siblings.

This is the "Dep placement" invariant in `.agents/skills/audit/SKILL.md`
("Main/preload deps stay in `dependencies`; renderer-only libs in
`devDependencies`"). Fixing it also makes the `architecture` skill's claim "the
shadcn next-themes wiring is stripped" actually true.

## Steps

### Step 1: Hardcode the sonner theme to dark, remove the `next-themes` import

In `src/renderer/src/components/ui/sonner.tsx`:
- Delete the line `import { useTheme } from "next-themes"`.
- Delete the line `const { theme = "system" } = useTheme()`.
- Change `theme={theme as ToasterProps["theme"]}` to `theme="dark"`.

The rest of the component (icons, style vars, `toastOptions`) is unchanged.

**Verify**: `grep -rn "next-themes" src/` → **no matches**.

### Step 2: Remove `next-themes` from the manifest

In `package.json`, delete the `"next-themes": "^0.4.6"` line from `dependencies`.

### Step 3: Move the jetbrains-mono font package to `devDependencies`

In `package.json`, remove `"@fontsource-variable/jetbrains-mono": "^5.2.8"` from
`dependencies` and add it to `devDependencies` (next to the two geist fonts, keep
the object alphabetically ordered as the surrounding entries are).

### Step 4: Reinstall to update the lockfile

Run `pnpm install`.

**Verify**: `pnpm install` → exit 0; `git diff --stat pnpm-lock.yaml` shows the
lockfile changed (next-themes removed).

### Step 5: Run the full gate

**Verify**: `pnpm verify` → lint, typecheck, test, build all exit 0.

The build is the important check here: it confirms the renderer still resolves all
three fonts from `devDependencies` (the two geist fonts already prove a
`devDependencies` fontsource package resolves via the CSS `@import`), and that
removing `next-themes` didn't break the bundle.
